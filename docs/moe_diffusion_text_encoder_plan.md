# 基于对比学习的 MoE Router-Only 微调方案文档

## 1. 文档目标

本文档只讨论一个收敛后的最小研究问题：

- 使用预训练 MoE 模型作为候选 text encoder；
- 训练时只微调 router；
- 训练目标只使用对比学习；
- 暂不讨论 diffusion 接入；
- 暂不讨论专家剪枝；
- 暂不讨论蒸馏、额外 head 或多目标联合训练。

换句话说，当前阶段要回答的不是“如何做完整系统”，而是：

- 仅通过 `router-only + contrastive learning`，能否让 MoE 的文本表征更适合目标 prompt 分布。

## 2. 问题定义

### 2.1 当前研究对象
- 输入是 diffusion 场景相关的 prompt 文本。
- 编码器主体是预训练 MoE 模型。
- 训练时冻结除 router 外的全部参数。
- 输出是用于表示 prompt 语义的文本向量或 token-level 隐表示。

### 2.2 当前研究目标
- 让相同语义或近似语义的 prompt 在表示空间中更接近。
- 让不同语义的 prompt 在表示空间中更可分离。
- 观察 router 是否会因为任务约束而产生更有结构的专家分配模式。

### 2.3 当前不解决的问题
- 训练后的编码器如何接入具体 diffusion backbone；
- 训练后的 token 表征是否已经足够支持 cross-attention；
- 专家剪枝后能否进一步压缩模型；
- 是否需要蒸馏到 CLIP/T5 风格表示空间。

## 3. 可行性判断

## 3.1 为什么这条路线成立的概率不低

### 理由 1：MoE 天然允许“按 token 重分配计算”
- prompt 中不同 token 往往承担不同语义角色，例如主体、属性、风格、关系、修饰词。
- router 的作用就是决定这些 token 走向哪些专家。
- 因此，router-only 微调本质上是在学习“prompt 的语义片段应该交给哪些专家处理”。

### 理由 2：对比学习适合先做表示空间重塑
- 当前阶段的目标不是生成，而是让文本表示先具备更强的任务相关区分能力。
- 对比学习在“拉近正样本、拉远负样本”这件事上足够直接，也容易解释。

### 理由 3：成本足够低，适合做第一阶段验证
- 只训练 router，可以大幅降低训练成本。
- 如果连这一阶段都不能带来稳定收益，那么后续更复杂的方案大概率也不值得继续。

## 3.2 主要风险

### 风险 1：router 调整能力有限
- router 只能改变专家调用路径，不能直接改变专家内部知识。
- 如果底座专家本身对目标 prompt 分布不敏感，收益可能有限。

### 风险 2：对比学习偏向全局语义
- 对比学习天然更偏句级别语义对齐。
- 它能证明“表示空间变好了”，但暂时不能直接证明“生成条件一定更好了”。

### 风险 3：MoE 底座的结构偏置
- 如果底座是 decoder-only MoE，那么它的隐藏状态未必天然适合作为 text encoder。
- 特别是在长 prompt、多属性绑定和组合关系方面，可能存在先天短板。

### 风险 4：路由塌缩
- 对比学习有可能让一部分专家被过度使用。
- 如果训练后大多数 token 都集中流向少数专家，说明模型没有学到健康的任务相关分工。

## 4. 方案边界

### 4.1 训练边界
- 只训练 router 参数。
- 其他参数全部冻结。
- 损失函数只保留对比损失。

### 4.2 模型边界
- 不引入额外 projection head。
- 不引入 teacher model。
- 不对底层专家做任何结构改动。

### 4.3 评测边界
- 只评估文本表征质量与路由行为变化。
- 当前不做生成质量评测。

## 5. 模型设计

## 5.1 底座模型要求

优先选择满足以下条件的 MoE 模型：

- router 模块边界清晰；
- 支持导出 token hidden states；
- 能方便冻结非 router 参数；
- 有稳定的开源实现；
- 训练成本可控。

## 5.2 编码输出形式

当前建议同时保留两类输出视角：

- 句级表示：用于对比学习主目标；
- token-level 隐表示：用于分析不同 prompt 在各层路由下的局部行为。

### 句级表示的推荐构造方式
- 优先使用最后一层 hidden states 做 masked mean pooling。
- 原因是它最简单，不会引入额外可训练参数。

## 5.3 参数冻结策略

### 原则
- `router` 相关参数可训练；
- 其余参数全部冻结。

### 这样做的意义
- 让实验结论更干净；
- 确保表征变化主要来自路由重分配，而不是底座本身被重写。

## 6. 对比学习方案

## 6.1 样本组织方式

每个训练样本至少包含一对文本：

- `anchor`：原始 prompt；
- `positive`：与原 prompt 语义等价或高度相近的改写。

负样本来自：

- 同一个 batch 内的其他样本；
- 或者人工构造的“语义接近但关键属性不同”的 prompt。

## 6.2 正样本构造策略

优先采用以下几类正样本：

- 同义改写；
- 词序变化但语义不变；
- 格式变化但语义不变；
- caption 重写；
- style token 重排但主语义不变。

## 6.3 负样本构造策略

优先采用以下几类负样本：

- 主体不同但模板相似；
- 主体相同但关键属性不同；
- 风格相同但内容主体不同；
- 关系词不同导致语义变化的 prompt。

## 6.4 损失函数

当前方案只使用标准对比损失：

- `L_total = L_contrastive`

推荐具体形式：

- 对称 InfoNCE；
- anchor 到 positive 一次；
- positive 到 anchor 再一次；
- 最终对两者求平均。

### 这样做的原因
- 实现简单；
- 对 batch 内负样本利用充分；
- 便于后续比较不同正负样本构造策略。

## 6.5 batch 设计建议

- batch 内 prompt 类型尽量多样；
- 不要让同一模板过度集中；
- 保留一部分长 prompt；
- 保留一部分多属性、多对象 prompt。

原因很直接：

- 如果 batch 过于单一，router 可能学到模板偏置，而不是语义分工。

## 7. 数据设计

## 7.1 数据来源建议

当前阶段只需要文本，不需要先处理图像：

- diffusion 社区常见 prompt；
- caption 数据中的文本部分；
- 人工改写的 prompt 对；
- 长 prompt 与组合 prompt 数据。

## 7.2 数据集最小格式

建议最小样本字段如下：

- `anchor_text`
- `positive_text`
- `group_id`
- `prompt_type`

其中：

- `group_id` 用来标记同一语义组；
- `prompt_type` 用来做后续分析，例如风格类、属性类、组合类。

## 7.3 数据覆盖重点

必须覆盖这些 prompt 现象：

- 风格描述；
- 主体对象；
- 属性词；
- 关系词；
- 长 prompt；
- 多对象组合；
- 属性绑定。

## 8. 训练流程

## 8.1 Step 1：建立冻结基线

目标：

- 不训练任何参数；
- 先观察原始 MoE 对目标 prompt 分布的表征效果；
- 记录原始路由分布。

输出：

- 基线表示质量；
- 不同 prompt 类型下的路由统计；
- 不同层的专家使用分布。

## 8.2 Step 2：router-only 对比学习微调

目标：

- 冻结非 router 参数；
- 只训练 router；
- 使用对比学习优化句级表示。

输出：

- 训练损失曲线；
- 训练前后句向量相似度结构变化；
- 训练前后路由分布变化；
- 不同 prompt 类型下的专家重分配现象。

## 8.3 Step 3：训练后分析

重点不是继续扩展系统，而是先回答三个问题：

1. 表示空间是否更适合目标 prompt 分布；
2. 路由模式是否产生了稳定变化；
3. 这种变化是否具有 prompt 类型上的可解释性。

## 9. 评测方案

## 9.1 表征质量评测

推荐使用以下指标：

- text-text retrieval；
- prompt 改写一致性；
- 类内紧致度；
- 类间分离度；
- 长 prompt 鲁棒性。

## 9.2 路由行为评测

当前阶段必须把路由分析当作正式指标，而不是附属可视化：

- 各层专家使用频率；
- 路由熵；
- top expert 占比；
- 不同 prompt_type 下的专家分布差异；
- 训练前后路由分布变化幅度。

## 9.3 成功标准

如果同时满足下面三点，可以认为当前阶段成立：

- 文本表征指标优于冻结基线；
- 路由模式发生稳定变化；
- 路由变化与 prompt 类型之间存在可解释关系。

## 10. 当前阶段的失败信号

如果出现以下情况，说明 `router-only + contrastive` 可能不够：

- 表征指标几乎没有改善；
- 路由分布几乎不变；
- 路由只集中到少数专家；
- prompt 类型和专家分配之间没有稳定关系。

## 12. 总结

当前最合理的推进方式不是马上做完整系统，而是先把问题收紧到：

- 一个 MoE 底座；
- 一个 router-only 微调策略；
- 一个对比学习目标；
- 一套文本表征与路由行为评测。

如果这一阶段成立，后续再考虑：

- diffusion 接入；
- 更复杂的目标函数；
- 专家剪枝；
- 结构扩展。

如果这一阶段都不能成立，继续增加系统复杂度通常只会掩盖问题，而不会真正解决问题。
